# Use case: Процесс - Роль - Документ - SLA

## Job to be done

Когда я проектирую изменение процесса, я хочу создать метамодель процесса с ролями, документами и SLA, чтобы одинаково хорошо обсуждать изменения с бизнесом, операцией и ИТ.

## Контекст

Пользователь - бизнес-аналитик или владелец процесса.

## Путь пользователя

1. Выбирает процесс, в котором реально есть боль: длительные согласования, потеря документов, неясные роли.
2. Выделяет обязательные сущности: шаг процесса, роль, документ, SLA, контрольная точка, KPI, pain point.
3. Создаёт типы объектов и короткие определения для каждого, чтобы все трактовали их одинаково.
4. Описывает связи: роль выполняет шаг, шаг создаёт документ, шаг ограничен SLA, шаг влияет на KPI.
5. Решает, какие элементы нужны на диаграмме, а какие лучше хранить как свойства.
6. Настраивает визуальный вид, который понятен бизнес-пользователю без архитектурной подготовки.
7. Заполняет модель по итогам одного-двух интервью с предметными экспертами.
8. Проверяет, можно ли по модели увидеть bottleneck, лишние согласования и документы без владельца.
9. Дополняет модель таблицей для реестра ролей и SLA.
10. Использует её как основу для согласования целевого процесса.

## Критерий успеха

Модель помогает обсуждать изменение процесса предметно и сокращает потери на пересказ между участниками.
