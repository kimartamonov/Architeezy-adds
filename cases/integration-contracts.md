# Use case: Интеграционный контракт

## Job to be done

Когда мне нужно быстро и одинаково описывать интеграции между системами, я хочу создать метамодель интеграционного контракта, чтобы фиксировать канал, формат, SLA, безопасность и владельцев в одном рабочем языке.

## Контекст

Пользователь - системный или интеграционный архитектор.

## Путь пользователя

1. Формулирует список решений, которые должна поддерживать модель: согласование нового интерфейса, ревью существующих контрактов, поиск ответственных.
2. Выделяет сущности первого уровня: система, контракт, канал, сообщение, схема, политика безопасности, SLA, команда.
3. Создаёт типы объектов и обязательные атрибуты для каждого из них.
4. Явно разделяет синхронные и асинхронные взаимодействия, чтобы не путать semantics в модели.
5. Описывает связи: поставщик публикует контракт, потребитель использует контракт, контракт использует схему, контракт имеет SLA и владельца.
6. Настраивает диаграммное представление для коммуникации между системами.
7. Настраивает табличное представление для инвентаризации контрактов.
8. Заполняет модель на примере одного продукта или домена.
9. Проверяет, что по модели можно ответить на вопросы "кто владелец", "какой формат", "какая политика retry", "какая аутентификация".
10. Добавляет соглашение по наименованию и версионированию контрактов.

## Критерий успеха

Каждая новая интеграция описывается по единому шаблону, а модель пригодна и для review, и для передачи в команды разработки.
