# Use case: Обмен данными и compliance

## Job to be done

Когда я должен описывать обмен данными с учётом PII, retention и маскирования, я хочу создать специализированную метамодель, чтобы видеть не только поток данных, но и обязательства по их безопасному использованию.

## Контекст

Пользователь - архитектор данных, архитектор интеграций или представитель data governance.

## Путь пользователя

1. Выбирает один чувствительный поток данных как пилотный кейс.
2. Создаёт типы для набора данных, поля, классификации чувствительности, основания обработки, срока хранения, маскирования, потребителя и контракта передачи.
3. Описывает обязательные атрибуты и жизненный цикл согласования обмена.
4. Настраивает связи между источником, набором данных, потребителем и мерами контроля.
5. Делает диаграммное представление для архитектурного review и табличный реестр полей/ограничений.
6. Заполняет модель по одному реальному интерфейсу.
7. Проверяет, можно ли быстро ответить, какие поля чувствительны, кто их получает и на каком основании.
8. Добавляет шаблон для новых обменов.
9. Согласует язык с ИБ и юристами, чтобы термины были корректными.
10. Использует модель как основу для регламентов и проверок.

## Критерий успеха

Требования compliance перестают теряться в текстах и становятся частью рабочей модели обмена данными.
