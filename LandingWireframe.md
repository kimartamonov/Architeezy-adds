# Wireframe лендинга Architeezy

## Назначение документа

Этот документ описывает рекомендуемую структуру лендинга Architeezy сверху вниз:

- какие блоки должны быть на странице;
- в каком порядке их лучше показывать;
- какую задачу решает каждый блок;
- какой ключевой смысл должен считать пользователь;
- какие CTA должны быть в каждом месте;
- какие визуальные элементы стоит заложить в дизайн.

Документ можно использовать как основу для:

- low-fidelity wireframe;
- прототипа в Figma;
- постановки дизайнеру;
- постановки фронтенд-разработчику;
- контентной сборки лендинга.

## Главная цель страницы

Провести пользователя по цепочке:

`это для меня -> это решает мою задачу -> я вижу, как это работает -> я понимаю, как стартовать -> я понимаю, какой вариант поставки мне подходит -> я делаю следующий шаг`

## Основной сценарий пользователя

1. Пользователь попадает на лендинг.
2. За 5-10 секунд понимает, для кого продукт и в чём его ценность.
3. Видит, что продукт решает не абстрактную, а практическую задачу.
4. Получает живое доказательство через демо, шаблоны и сценарии.
5. Понимает, как начать: SaaS, on-premise или enterprise.
6. Кликает по одному из ключевых CTA.

## Глобальная логика страницы

1. Hero.
2. Проблема рынка.
3. Решение и ключевые выгоды.
4. Быстрый путь к первой пользе.
5. Демо и доказательства.
6. Сценарии по ролям.
7. Security и deployments.
8. Сравнение вариантов поставки.
9. Кейсы и доверие.
10. FAQ.
11. Roadmap.
12. Финальный CTA.

## Блок 1. Hero

### Задача блока

Сразу объяснить, что Architeezy это не просто редактор схем, а единая рабочая среда для архитектуры, документации и моделей.

### Что должен понять пользователь

- продукт предназначен для архитекторов, аналитиков и инженерных команд;
- это единый репозиторий моделей и документации;
- можно начать через демо, облако или on-premise.

### Контент

- заголовок;
- подзаголовок;
- короткая строка с нотациями и DSL;
- 3 CTA;
- короткий supporting line про SaaS / on-premise / enterprise.

### CTA

- `Открыть демо`
- `Начать бесплатно`
- `Развернуть локально`

### Визуально

- крупный headline;
- справа или ниже визуал интерфейса;
- можно показать переключение между диаграммой, таблицей и документом;
- рядом 3 коротких use case pills.

### Что важно в дизайне

- первый экран должен быть очень читаемым;
- не перегружать hero длинным списком фич;
- CTA должны быть видны без скролла.

## Блок 2. Почему просто диаграмм недостаточно

### Задача блока

Активировать боль и показать, что пользователь сталкивается не с проблемой "как нарисовать", а с проблемой "как поддерживать архитектуру живой и согласованной".

### Что должен понять пользователь

- схемы, таблицы и документы по отдельности создают хаос;
- проблема знакомая и системная;
- Architeezy решает именно этот разрыв.

### Контент

- короткий заголовок о боли;
- 3-4 карточки с типовыми проблемами;
- короткий переход к решению.

### Визуально

- можно сделать контрастный блок;
- слева проблема "разрозненные артефакты", справа "единая модель";
- допустима схема "до / после".

## Блок 3. Что получает команда

### Задача блока

Показать ключевые выгоды на уровне результата, а не каталог функций.

### Что должен понять пользователь

- одна модель может обслуживать много сценариев;
- продукт закрывает совместную работу, представления, публикацию, DSL и автоматизацию;
- это платформа, а не узкий редактор.

### Контент

6 карточек:

- единый источник истины;
- разные представления модели;
- совместная работа;
- нотации и DSL;
- API / AI / MCP;
- документация из модели.

### Визуально

- сетка из карточек;
- иконки или маленькие скриншоты;
- короткие и очень считываемые подписи.

## Блок 4. Первая победа за 10 минут

### Задача блока

Сделать старт психологически лёгким и показать понятный путь первого результата.

### Что должен понять пользователь

- старт не выглядит тяжёлым;
- первый результат достижим быстро;
- есть путь и для знакомства, и для пилота, и для on-premise.

### Контент

4 шага:

1. создать проект;
2. выбрать шаблон или нотацию;
3. собрать первую модель;
4. поделиться или выбрать сценарий поставки.

### CTA

- `Открыть демо`
- `Начать бесплатно`

### Визуально

- timeline или stepper;
- желательно с маленькими скриншотами или пиктограммами.

## Блок 5. Демо и шаблоны

### Задача блока

Дать живое доказательство, что продукт уже работает и подходит для реальных сценариев.

### Что должен понять пользователь

- можно посмотреть продукт без долгого онбординга;
- есть прикладные шаблоны;
- ценность можно увидеть сразу.

### Контент

- заголовок;
- короткое описание;
- карточка публичного демо;
- 4-6 карточек шаблонов;
- мини-guide "что посмотреть в демо".

### CTA

- `Открыть демо-проект`
- `Смотреть шаблоны`

### Визуально

- превью демо;
- сетка шаблонов;
- опционально короткое видео или gif.

## Блок 6. Сценарии по ролям

### Задача блока

Поднять релевантность для разных сегментов аудитории.

### Что должен понять пользователь

- продукт подходит не только "архитектору вообще";
- есть конкретные сценарии для моей роли;
- разные роли работают с одной моделью по-разному.

### Контент

4 карточки ролей:

- enterprise architect;
- solution / integration architect;
- analyst / process owner;
- platform / infrastructure team.

У каждой карточки:

- типовая задача;
- результат;
- ссылка на сценарий или демо.

### Визуально

- горизонтальный ряд или сетка;
- можно использовать отдельные цветовые акценты по ролям.

## Блок 7. Security и deployments

### Задача блока

Закрыть ключевую B2B-тревогу по безопасности, контролю данных и формату внедрения.

### Что должен понять пользователь

- есть не только SaaS;
- on-premise и enterprise являются нормальными сценариями;
- чувствительные данные можно вести в более контролируемом контуре.

### Контент

3 карточки:

- SaaS;
- on-premise;
- enterprise.

Под ними:

- короткие тезисы по контролю данных;
- доступам;
- IAM / SSO;
- поддержке внедрения.

### CTA

- `Сравнить варианты поставки`
- `Развернуть локально`
- `Запросить внедрение`

### Визуально

- спокойный, более "корпоративный" блок;
- можно использовать таблицу или три вертикальные карточки.

## Блок 8. Сравнение SaaS / On-premise / Enterprise

### Задача блока

Помочь выбрать правильный сценарий без лишней неопределённости.

### Что должен понять пользователь

- какой вариант подходит под его контекст;
- чем отличаются скорость старта, контроль и поддержка;
- куда кликнуть дальше.

### Контент

Таблица сравнения по строкам:

- скорость старта;
- контроль данных;
- IAM / SSO;
- поддержка;
- SLA;
- кому подходит;
- как начать.

### CTA

- `Начать бесплатно`
- `Развернуть локально`
- `Запросить enterprise`

### Визуально

- чистая и простая comparison table;
- выделить рекомендуемый сценарий для каждого типа задачи через подписи, а не через агрессивный sales-style.

## Блок 9. Кейсы и доверие

### Задача блока

Поднять доверие за счёт прикладных кейсов и social proof.

### Что должен понять пользователь

- продукт применим в реальной работе;
- им можно решать разные архитектурные сценарии;
- у платформы есть зрелость и практическая ценность.

### Контент

- 3-4 карточки кейсов;
- логотипы или цитаты, если есть;
- факты и цифры, если есть;
- ссылка на полную страницу кейсов.

### CTA

- `Смотреть кейсы`
- `Открыть демо`

### Визуально

- карточки кейсов;
- блок логотипов;
- короткие цитаты.

## Блок 10. FAQ

### Задача блока

Снять типовые возражения прямо на лендинге и не отправлять пользователя искать ответы самостоятельно.

### Что должен понять пользователь

- основные вопросы по старту, безопасности, API, нотациям и контуру уже продуманы;
- переход на следующий шаг безопасен и понятен.

### Контент

8-10 вопросов:

- это просто редактор диаграмм?;
- для кого подходит?;
- какие нотации есть?;
- можно ли сделать свой DSL?;
- есть ли API?;
- можно ли развернуть локально?;
- что делать с чувствительными данными?;
- что входит в бесплатный старт?;
- когда нужен enterprise?;
- можно ли экспортировать / публиковать?

### Визуально

- accordion;
- каждая категория вопросов может быть сгруппирована: старт, продукт, безопасность, интеграции.

## Блок 11. Roadmap

### Задача блока

Показать развитие продукта без ощущения, что основная ценность всё ещё "в будущем".

### Что должен понять пользователь

- продукт развивается;
- уже сейчас доступно много полезного;
- roadmap усиливает доверие, а не отвлекает от текущей ценности.

### Контент

- короткий блок `Что доступно уже сейчас`;
- roadmap в свернутом виде;
- ссылка `Посмотреть планы развития`.

### Визуально

- компактно;
- не выше по странице, чем блоки доверия и выбора сценария;
- не давать roadmap визуально конкурировать с demo и CTA.

## Блок 12. Финальный CTA

### Задача блока

Поймать пользователя после прохождения всей страницы и дать три понятных направления действия.

### Что должен понять пользователь

- путь к следующему шагу прост;
- выбор зависит от сценария;
- ему не нужно "додумывать", куда идти.

### Контент

#### Заголовок

Выберите удобный путь старта

#### Подзаголовок

Откройте демо для быстрого знакомства, начните бесплатно для пилота или выберите on-premise / enterprise для контролируемого внедрения.

#### CTA

- `Открыть демо`
- `Начать бесплатно`
- `Запросить внедрение`

## Sticky-элементы и навигация

### Что стоит предусмотреть

- sticky header;
- якоря на разделы:
  - Возможности
  - Демо
  - Сценарии
  - Security
  - FAQ
- компактная кнопка `Начать бесплатно` и `Открыть демо` в шапке.

## Мобильная версия

### Требования

- hero должен быстро читаться без перегруза;
- CTA в первом экране должны помещаться удобно;
- таблицу сравнения нужно адаптировать в карточки или горизонтальный скролл;
- FAQ и roadmap должны быть компактными;
- демо-превью не должно ломать первый сценарий скролла.

## Приоритет для первой версии

Если делать сокращённую первую итерацию лендинга, минимально обязательны:

1. Hero.
2. Блок проблемы.
3. Блок выгод.
4. Первая победа за 10 минут.
5. Демо.
6. Security / deployments.
7. FAQ.
8. Финальный CTA.

## Что нужно дизайнеру на старте

- copy из `LandingCopyDraft.md`;
- список страниц и задач из `LandingTasks/`;
- этот wireframe;
- понимание, что лендинг должен быть одновременно:
  - понятным для нового пользователя;
  - убедительным для B2B-аудитории;
  - достаточно "живым", чтобы не выглядеть тяжёлым enterprise-сайтом.

## Что нужно разработчику на старте

- структура блоков сверху вниз;
- список CTA и путей перехода;
- понимание, какие блоки могут вести на отдельные страницы;
- приоритет первой сборки и будущих A/B-тестов.
