# Создать страницу: интеграции, API и технические сценарии

## Что нужно создать

Отдельную страницу для технической аудитории, где REST API, AI и MCP объясняются через реальные примеры, а не через общие тезисы.

## Ключевая задача

Повысить доверие технической аудитории и показать, что Architeezy можно встроить в процессы, а не использовать только как визуальный инструмент.

## Ключевая аудитория

- solution architect;
- integration architect;
- platform engineer;
- техлид;
- инженерные команды, оценивающие расширяемость платформы.

## Главные вопросы, которые страница должна снять

- Что можно сделать через API на практике?
- Можно ли связать Architeezy с внутренними процессами и инструментами?
- Как AI и MCP помогают именно в работе с моделями?
- Можно ли автоматизировать реестры, документы и обновления модели?

## Главные возражения, которые страница должна снять

- "REST API заявлен, но непонятно, насколько он полезен."
- "AI/MCP звучат модно, но неясно, как это применять."
- "Не вижу сценария, как встроить платформу в существующий контур."

## Что должно быть на странице

### 1. 2-3 прикладных сценария

Например:

- создать или обновить реестр через API;
- выгрузить данные модели в документ или таблицу;
- использовать AI/MCP для черновика модели по текстовому описанию;
- встроить модель в governance-review процесс.

### 2. Формат каждого сценария

- входная задача;
- что делает пользователь;
- что автоматизируется;
- какой результат получает команда.

### 3. Технические артефакты

- короткие примеры запросов;
- схема потока данных;
- ссылка на документацию;
- ссылка на self-hosted / security-страницу, если это важно.

### 4. Блок "Кому это полезно"

- архитекторы;
- platform / automation teams;
- команды enterprise-интеграции;
- консультанты и внутренние платформенные команды.

### 5. CTA

- `Смотреть документацию`;
- `Открыть демо`;
- `Обсудить enterprise-внедрение`.

## Контентные требования

- Никаких чисто декларативных формулировок без примеров.
- Даже сложные технологии объяснять через практический workflow.
- Показывать не только "можно", но и "зачем это делать".

## Желаемый результат

Технический посетитель должен увидеть, что платформа подходит для автоматизации и расширяемых процессов, а не только для визуального моделирования.
