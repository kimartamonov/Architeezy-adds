# Создать страницу: безопасность и варианты развертывания

## Что нужно создать

Отдельную страницу `Security & Deployments`, на которую ведут ссылки с главной, enterprise-страницы и сравнения пакетов.

## Ключевая задача

Перевести тему безопасности из фона тревоги в понятный аргумент выбора правильного сценария: SaaS, on-premise или enterprise.

## Ключевая аудитория

- enterprise architect;
- security / compliance;
- platform / infrastructure architect;
- техлид или руководитель, оценивающий риски внедрения.

## Главные вопросы, которые страница должна снять

- Где хранятся данные?
- Какие есть варианты контроля окружения?
- Что доступно в SaaS, а что лучше решать через on-premise?
- Есть ли SSO / IAM / Keycloak / роли / доступы?
- Как устроены резервирование, обновления, сетевой контур, reverse proxy, HTTPS?

## Главные возражения, которые страница должна снять

- "Для корпоративных данных облако слишком рискованно."
- "Не вижу, как закрываются требования по контролю доступа."
- "Неясно, можно ли развернуть продукт в своём контуре."
- "Лендинг говорит про безопасность слишком общо."

## Что должно быть на странице

### 1. Блок "Варианты развертывания"

Отдельно и ясно:

- SaaS;
- on-premise;
- enterprise.

Для каждого:

- где находятся данные;
- кто управляет окружением;
- кому подходит;
- когда этот вариант выбирать.

### 2. Блок "Контроль доступа и идентификация"

Раскрыть:

- роли и права;
- SSO / IAM;
- Keycloak или аналоги;
- сценарии для корпоративной аутентификации.

### 3. Блок "Операционная безопасность"

Описать:

- HTTPS;
- reverse proxy;
- секреты;
- резервное копирование;
- обновления;
- закрытый контур;
- аудит и контроль изменений, если доступно.

### 4. Блок "Для чувствительных данных"

Очень чётко сформулировать:

- для чувствительных данных рекомендуется on-premise / enterprise;
- это не ограничение, а понятный путь для компаний с высокими требованиями.

### 5. Блок с документами и ссылками

- документация по self-hosted;
- инструкция по развертыванию;
- контакты для enterprise-запроса.

### 6. CTA

- `Развернуть локально`;
- `Запросить SLA / внедрение`;
- `Сравнить варианты поставки`.

## Контентные требования

- Писать более формально и "закупочно-понятно", чем на общей продуктовой странице.
- Убирать двусмысленность.
- Не спорить с рисками, а показывать правильный маршрут для каждого типа данных и требований.

## Желаемый результат

Страница должна повысить доверие у B2B-аудитории и помочь посетителю самому выбрать подходящий вариант поставки вместо ухода с сайта из-за сомнений по безопасности.
