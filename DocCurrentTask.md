# Мастер-промт: реализуй текущий документ по журналу релиза

Ты специалист по созданию полезной, понятной и практически применимой документации для пользователей Architeezy.

Твоя задача: за один проход реализовать текущий документ, назначенный в `docReleaseJournal.md`, и обновить журнал так, чтобы следующий запуск команды `Реализуй DocCurrentTask.md` автоматически взял следующую страницу.

## Главный принцип

Всегда работай не "по памяти", а по текущему состоянию файлов проекта.

Перед началом обязательно прочитай:

1. `docReleaseJournal.md`
2. `meta-doc-structure.md`
3. файл требований, указанный в разделе `Текущая задача` журнала
4. при необходимости связанные сценарии из `cases/`
5. при необходимости связанные потребности из `doc-needs/`

## Что нужно сделать

### Шаг 1. Найти текущую задачу

В `docReleaseJournal.md` найди раздел `Текущая задача`.

Используй только те значения, которые прямо указаны там:

- файл требований;
- целевой файл документа;
- файл отчёта;
- порядок и название страницы.

Если в журнале написано, что текущих задач больше нет, ничего нового не создавай и только зафиксируй, что очередь завершена.

### Шаг 2. Определить, для кого пишется текущий документ

Определи аудиторию не по догадке, а по данным из:

- файла требований в `metadoc-req/`;
- связанных сценариев из `cases/`;
- связанных потребностей из `doc-needs/`.

При определении уровня пользователя учитывай:

- роль пользователя;
- его типичную практическую задачу;
- уровень технической подготовки;
- глубину понимания моделирования.

Используй такие ориентиры:

- enterprise и solution архитекторы: уверенный профессиональный уровень, можно писать содержательно и структурно, но без лишней академичности;
- бизнес-аналитики и владельцы процессов: средний технический уровень, упор на ясность, примеры и прикладные шаги;
- data stewards и data governance пользователи: смешанный уровень, нужно объяснять и бизнес-смысл, и технические последствия;
- инфраструктурные и платформенные архитекторы, SRE: высокий технический уровень, но всё равно без перегруженного теоретического жаргона.

Не пиши слишком сложно, но и не упрощай ниже реального уровня понимания целевого пользователя.

## Шаг 3. Написать сам документ

Создай или обнови целевой файл документа из журнала.

Документ должен:

- быть полезным пользователю в реальной работе;
- опираться на требования из файла `metadoc-req/...`;
- закрывать вопросы пользователя, перечисленные в требованиях;
- быть написанным как пользовательская документация, а не как внутренняя заметка;
- вести читателя от понимания к действию;
- не пересказывать требования буквально, а превращать их в полноценную страницу документации.

## Требования к качеству документа

Документ должен:

- объяснять, когда и зачем пользователю нужен данный материал;
- учитывать реальный контекст ролей и сценариев;
- содержать понятные определения и пошаговые рекомендации там, где это нужно;
- использовать примеры, близкие к сценариям из `cases/`;
- показывать типичные ошибки, ограничения и хорошие практики, если это уместно;
- помогать пользователю понять, что делать следующим шагом;
- быть структурированным и удобным для чтения.

## Рекомендуемая форма страницы

Адаптируй структуру под тип страницы, но обычно страница должна включать большую часть таких блоков:

- о чём эта страница;
- когда она полезна;
- для кого она написана;
- ключевая идея;
- пошаговый подход или правила проектирования;
- примеры для реальных сценариев;
- типичные ошибки и антипаттерны;
- практический чек-лист или критерий готовности;
- что читать дальше.

Если страница является прикладным паттерном, делай упор на:

- типовой состав сущностей;
- атрибуты и связи;
- рекомендуемые представления;
- ограничения и риски;
- адаптацию паттерна под разные похожие сценарии.

Если страница является методической, делай упор на:

- ход рассуждения пользователя;
- критерии принятия решений;
- правила выбора между вариантами;
- практические признаки правильного и неправильного подхода.

## Шаг 4. Создать отчёт о выполнении

После написания документа создай файл отчёта по пути из журнала.

Отчёт должен содержать:

- какой документ был реализован;
- какие исходные файлы использовались;
- для каких ролей и сценариев был написан документ;
- какие ключевые темы вошли в страницу;
- какие допущения были сделаны;
- какие риски или открытые вопросы остались;
- какой следующий документ должен быть реализован дальше.

Отчёт должен быть кратким, но полезным для контроля прогресса.

## Шаг 5. Обновить журнал `docReleaseJournal.md`

После успешной реализации текущего документа обязательно обнови журнал.

Нужно сделать всё сразу:

1. В разделе `Реализованные документы` добавить новую запись с датой, порядком, названием страницы, путём к документу и путём к отчёту.
2. В таблице `Очередь документов` поменять статус текущей строки с `CURRENT` на `DONE`.
3. Найти следующую строку со статусом `QUEUED` и поменять её статус на `CURRENT`.
4. Обновить раздел `Текущая задача` вверху журнала так, чтобы он указывал уже на следующую страницу.
5. Если завершён последний документ, явно написать в разделе `Текущая задача`, что очередь завершена.

## Правила обновления журнала

- Не удаляй историю уже реализованных документов.
- Не меняй порядок очереди.
- Не пропускай задачи.
- Не меняй пути к файлам без необходимости.
- Если документ уже существует и ты его дорабатываешь, всё равно журнал должен отражать реальное текущее состояние очереди.

## Ожидаемый результат одного запуска

После одного запуска должны существовать и быть обновлены три вещи:

1. целевой документ в `metadoc-pages/`;
2. отчёт о его реализации в `metadoc-reports/`;
3. обновлённый `docReleaseJournal.md`, указывающий на следующий документ.

## Команда использования

Каждый следующий запуск выполняется одной и той же командой:

`Реализуй DocCurrentTask.md`
